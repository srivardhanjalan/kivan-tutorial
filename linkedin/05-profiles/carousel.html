What the step shipped

One screen, two hard problems.

The Settings screen — change your name, set your birthday, delete your account — took an afternoon. The real work was underneath: keeping your name in sync, and deleting you so you're actually gone.
Handle With Care2 / 8
The bug I inherited

Write to every store that reads.

The obvious way to save a name is to write it to Clerk. But the backend keeps its own copy — written once at sign-in, create-only. Write to Clerk alone and it goes stale, with nothing to heal it.
// Clerk greets you; the backend record is the copy later steps read
await user?.update({ firstName, lastName });
await updateProfile({ first_name, last_name });
A fact kept in two places, edited in one, is a bug with a delay on it.
Handle With Care3 / 8
Padding is data

The wheels came up blank.

The birthday is three picker wheels. Values were unpadded ("1"), labels padded ("01") — so a stored "01" matched no wheel value. The fix strips padding on load, re-pads on save.
Handle With Care4 / 8
Deleting you properly

Three decisions, each easy to botch.

→ Keep the record, just flag it — later steps point at it.

→ Delete from Clerk first, then flag — so whichever write fails, the leftover state is the harmless one.

→ Guard the write at the database, not in app memory — a cached instance could keep writing after you're gone.
Handle With Care5 / 8
The guard

At the table, not in memory.

The flag write itself is unconditional. Every mutating endpoint carries a condition DynamoDB re-checks — the record must exist and not be flagged deleted. A token minted seconds before deletion bounces.
ACTIVE = "attribute_exists(id) AND (attribute_not_exists(is_deleted) OR is_deleted = :active)"
users.update_item(Key=..., ConditionExpression=ACTIVE) # 403 on a dead token
